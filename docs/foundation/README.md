# Foundation

> **Looking for the full map of Aurora's intellectual architecture?** Start at the canonical [Foundation Index](../README.md), which connects all seven discovery papers (foundation, domain, and architecture) in reading order.

The Foundation documents establish why Aurora exists and what it stands for.

They are not technical documentation. They are not product specs. They do not describe what Aurora does.

They describe what Aurora *believes* — and why those beliefs make Aurora necessary.

## The role of Foundation documents

Every product accumulates decisions over time. Features are added. Priorities shift. Teams change. Without a stable foundation, those decisions become disconnected from each other, and the product drifts.

Foundation documents exist to prevent that drift.

They are not rigid specifications. They are a shared language for making decisions — a set of beliefs the team can return to when the path is unclear.

When a feature proposal lands on the table, the question is not "can we build this?" The question is "does this belong here?" The Foundation documents are the basis for that question.

---

## How Foundation documents differ from other docs

| Type | Answers | Changes when |
|---|---|---|
| **Foundation** | *Why Aurora exists* | Core beliefs about the problem change — rarely |
| **Product** (`docs/product/`) | *What Aurora does and for whom* | Product strategy evolves |
| **Domain** (`docs/domain/`) | *What concepts Aurora works with* | Domain understanding deepens |
| **Architecture** (`docs/architecture/`) | *How Aurora is built* | Technical decisions change |
| **Specs** (`docs/specs/`) | *What each feature does* | Features are added or changed |

Foundation is the most stable layer. If the Foundation changes, it is a significant event — not a routine update.

## How to read THE_AURORA_THESIS

Read it once, completely, before contributing to Aurora in any role.

Then return to it when a decision is contested.

The Thesis is not a document to reference for technical guidance. It is a document to consult when the team needs to know what Aurora is *for*.
